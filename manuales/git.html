GIT

En este manual revisaremos conceptos basicos, asi como recomendaciones a la hora de desarrollar proyectos informaticos.

Conceptos previos

Git es un sistema de versionamiento de codigo que permite trabajar de manera ordenada un proyecto informatico, manteniendo un historico y ramas (branches) para generar versiones del proyecto.

Para empezar a trabajar se recomienda usar un servidor GIT (no es necesario) ademas de saber las diferentes maneras de conectarnos.

Servidor Git

Para poder usar Git es recomendable usar un servidor central, que almacene el codigo. Existen servidores gratuitos pero Github es el mas popular.

Para conectar con el servidor existen varios metodos, los mas usados son HTTPS y ssh. La mas rapida es la coneccion via SSH. Mas adelante detallaremos como trabajar con cada protocolo.

Terminos usados

Branch : Rama de un proyecto informatico, se puede tener varias ramas para mantener todo ordenado (master, devel, release, etc.). Con GIT es posible mantener ramas autonomas e intercambiar uno del otro sin problemas.

Tag: Es una indicador en el proyecto, un hito importante que permite identificar un punto en la historia, por ejemplo el lanzamiento de la version 1.0.

Commit : Conjunto de cambios con su descripcion que seran subidos al versionamiento de codigo, esta es la parte fundamental del versionamiento. GIT identificada cada commit con un alfanumero aleatorio.


Primeros Pasos

Para inicar con GIT debemos tener instalado, podemos ir a la pagina oficial para descargar de acuerdo a nuestro sistema operativo. Luego de ello podemos usar la linea de consola para trabajar o algun cliente en especifico, en este caso nos enfocaremos en la linea de comantos.

Iniciar un proyecto

Si ya tenemos un proyecto informatico y queremos empezar a versionar el codigo necesitamos inicializar el GIT en dicha carpeta, para ellos nos dirigimos al proyecto y en la consola ingresamos:

git init

Se crea una carpeta oculta .git indicando que el archivo esta versionado, ademas se crea por defecto el branch master donde subiremos todos los cambios.

Para realizar nuestro primer commit, realizamos un cambio en nuestro proyecto, luego para ver los cambios usaremos :

git status

Nos retorna la lista de archivos cambiados, luego de ello agregamos los archivos para ser commiteados :

git add --all

Y procedemos a realizar el commit :

git commit -m 'Primer commit en GIT'

Trabajar con un Servidor

Siempre es necesario realizar almacenar nuestros commits en un servidor GIT para luego tener una copia de respaldo y trabajar de manera mas segura. Existen varios servidores GIT, explicaremos usando Github como servidor.

Si el proyecto esta en el servidor, es necesario realizar un clon del proyecto en nuestro entorno de desarollo local, para ello podemos usar diferentes protocolos (solo veremos el https y ssh).

Para usar https necesitaremos nuestra cuenta Github (usuario y password)

git clone https://usuario@github.com/usuario/proyecto.git

En el caso que usemos ssh (para esto necesitaremos asociar una llave a nuestra cuenta):

git clone git@github.com:usuario/proyecto.git

Luego de ello podemos realizar cambios y hacer los commits realizados (explicados anteriormente) pero aqui existen dos nuevos conceptos pull y push el cual sirve para "jalar" y "subir" los cambios respectivamentes al servidor GIT.

Si es que tenemos nuestro proyecto en nuestro entorno local, pero hay varias personas trabajando en el mismo es necesario tener una copia actualizada, para no generar conflictos a la hora de subir nuestros cambios, asi que lo primero que se recomienda es jalar los cambios.

git pull origin master

Con este comando jalamos los cambios del branch master del servidor (origin), ahora una vez realizado los cambios y commiteados los mismos, procedemos a subirlos al servidor:

git push origin master

Asi podemos ir trabajando de manera colectiva, manteniendo un proyecto integrado y manejando los cambios.


Comandos Adicionales

git branch rama cambia de rama en el proyecto, de master a devel o release, todo el codigo cambio, pues cada rama es autonoma. Es importante realizar un commit con los cambios hechos sino existiran problemas al realizar el cambio.

git log nos permite ver el historico de commits realizados en el repositorio.

git blame ruta/del/archivo con esto podemos verificar quien hice que cambios en un archivo especifico.


Recomendaciones

Para colaborar con los proyectos del Equipo de Desarrollo del Nucleo, es necesario cumplir una seria de estandares que permita mantener el codigo ordenado.

Todo cambio rutinario se realiza en el branch devel, cuando se tenga un proyecto estabilizado el codigo sera transferido al branch release para realizar pruebas, luego de superadas las pruebas se usa master es decir el proyecto mas estable sera master.

Todo usuario tendra configurado su usuario de github para poder realizar cambios en el servidor, si no lo hace, sera rechazado. Para configurar el usuario de git realizamos en la consola:

git config --global user.name "Firstname Lastname"

git config --global user.email "your_email@youremail.com"
